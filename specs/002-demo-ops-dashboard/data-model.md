# Data Model: Demo Operations Dashboard

## DemoScenario

Represents a deterministic hackathon presentation scenario.

**Fields**:
- `scenario_id`: stable scenario identifier, e.g. `hackathon-main`.
- `name`: display name.
- `loaded_at`: timestamp when scenario was loaded.
- `status`: `empty`, `loaded`, `running`, `paused`, `reset`.
- `simulation_interval_seconds`: presenter-configured live update interval.
- `active_step`: current simulation step number.
- `message`: latest presenter-facing confirmation message.

**Relationships**:
- Has many `DemoShipmentState` records.
- Has many `SimulationStep` records.
- Has many `PresenterControlEvent` records.
- Has many `ManagerDecisionEvent` records.

## DemoShipmentState

Dashboard-ready state for one demo shipment, derived from existing shipment,
telemetry, assessment, and recommendation records.

**Fields**:
- `shipment_id`, `trailer_id`, `truck_id`, `pallet_id`.
- `crop`, `origin`, `planned_destination`, `recommended_destination`.
- `risk_level`: `healthy`, `watch`, `at_risk`, `critical`, `manual_review`,
  `reroute_approved`, `rejected`.
- `pallet_color`: `green`, `yellow`, `red`, `gray`.
- `remaining_shelf_life_hours`.
- `remaining_shelf_life_percent`.
- `temperature_c`.
- `safe_temp_min_c`, `safe_temp_max_c`.
- `temperature_history`: ordered typed readings for charting.
- `estimated_value_usd`.
- `value_protected_usd`.
- `time_remaining_to_act_minutes`.
- `confirmation_message`.

**Validation rules**:
- Percent fields must stay between 0 and 100.
- Pallet color must be derived from risk level and telemetry state.
- Demo data must be fictional.

## SimulationStep

Represents one live or manual simulation tick.

**Fields**:
- `step_id`: unique identifier.
- `scenario_id`: parent scenario.
- `step_number`: monotonically increasing integer.
- `mode`: `live` or `manual`.
- `created_at`: timestamp.
- `generated_readings`: typed telemetry readings generated by the step.
- `changed_shipments`: shipment IDs whose state changed.
- `message`: presenter-facing summary of the step.

## PresenterControlEvent

Represents a manual demo control.

**Fields**:
- `event_id`: unique identifier.
- `scenario_id`.
- `shipment_id`: optional target shipment.
- `control`: `load_demo`, `start_live`, `pause`, `advance_step`,
  `cooling_failure`, `temperature_spike`, `restore_cooling`,
  `telemetry_outage`, `reset`.
- `created_at`.
- `prior_state`.
- `resulting_state`.
- `message`.

## ManagerDecisionEvent

Represents an interactive manager decision from the demo dashboard.

**Fields**:
- `event_id`: unique identifier.
- `scenario_id`.
- `shipment_id`.
- `action`: `approve_reroute`, `send_manual_review`, `reject_shipment`.
- `actor_label`: presenter or manager display label.
- `created_at`.
- `prior_status`.
- `resulting_status`.
- `message`.

## State Transitions

```text
empty -> loaded -> running -> paused -> running
   ^        |          |        |
   `-------- reset <---`--------`
```

Shipment risk transitions:

```text
healthy -> watch -> at_risk -> critical -> reroute_approved
                         |          |          |
                         |          `-> rejected
                         `-> manual_review <- telemetry_outage
```

**Rules**:
- Load Demo Scenario creates exactly four seeded shipments with distinct initial
  risk levels.
- Cooling failure gradually worsens temperature and shelf-life state.
- Severe temperature spike can move a shipment directly to critical.
- Restore cooling stops worsening temperatures but does not erase prior damage.
- Telemetry outage maps the shipment to manual review/degraded visual state.
- Reset clears demo-specific scenario, simulation, and decision state, then
  returns the dashboard to the helpful empty state.
